# Composed rearchitecture — verified design (composition, not inheritance)

Adversarially-verified by the design workflow (wf_b36647c7). The seam/parity/migration
skeptics each tried to break it; the rules below are what survived.

> CORRECTION found during implementation: MOVEABLE_SLEEP already settled-skips the
> move pass (build.sh:3542). So the move-phase cull (old Step 1) is largely redundant;
> the first VALUABLE system is the RENDER cull (don't draw the ~300 occluded deck
> cards) — that's where the 25ms draw + 1276 draw-calls/frame live. Migration reordered

## composed core

ENTITY = the existing Card object (keyed by Card.sort_id / Card.ID), already registered in G.I.CARD and G.MOVEABLES. We do NOT introduce a parallel SoA entity store in phase 1 — the skeptic proved that "views over Card fields" + "swap component data" cannot coexist (code.lua:2577 contradiction: swap-over-views = use-after-free of T/VT; own-data reintroduces the dual-writer of T at cardarea.lua:561 + the eased VT writer at moveable.lua:404-419). RESOLUTION: components are split by WHO ALREADY WRITES the field, and the core OWNS only fields with exactly ONE legacy writer (or zero).

COMPONENTS (three ownership classes, this distinction is the whole design):
1. OWNED-NEW (core is sole writer, no legacy writer exists): `Active{visible:bool, dirty:bool, last_render_key}`. Pure additive metadata bolted onto Card. Zero existing code writes it. This is the only class the first slice touches.
2. VIEWED-RO (core reads, legacy is sole writer): `Render{VT.x,VT.y,VT.r,VT.scale}` (eased render-truth, written ONLY by Moveable:move_xy/_r/_scale at moveable.lua:404-454) and `Sprite{children.center,front,back,floating_sprite}` (the welded Glued/Minor DAG). Core NEVER writes these — it reads VT to decide active-set membership and render batching. No aliasing hazard because there is one writer.
3. CONTESTED (two legacy writers — quarantined, NOT a component until its writers are unified): `Layout{T.x,T.y,T.r}` has TWO writers — CardArea:align_cards (cardarea.lua:561-579) and Moveable:move/glue. This is the field the skeptic weaponized. It stays legacy-owned through the entire migration; the Layout system (step 3) does not OWN T, it REPLACES align_cards' sort+rank with a dirty-gated equivalent that writes the identical T.

PIPELINE = a thin scheduler invoked FROM INSIDE the existing per-frame loops, not as a sibling of G:update. Systems run in fixed order over the ACTIVE SLICE only (Active.visible cards), each a pure function (read components, write its owned component): ActiveSet(100, computes Active.visible/dirty from VT+states) -> Layout(200, dirty-gated align) -> Effect(400, event-driven over subscription index) -> Render(500, batched draw of active slice). Move(300) is NOT a new system — it remains Moveable:move; the pipeline reads its VT output. The core is `ecs/world.lua` (component arrays as parallel tables keyed by Card), `ecs/pipeline.lua` (ordered system registry: explicit data, one delete removes a system, no system owns the next call), `ecs/components.lua` (the three ownership classes above with per-field sync rules), `ecs/systems/{activeset,layout,effect,render}.lua`.

## the seam

THE SEAM IS NOT main.lua:1050/1056. The seam-skeptic is correct: love.update->G:update is already 5-deep wrapped (SMODS/_/src/ui.lua:98 `local gameUpdateRef=Game.update; function Game:update` ... ; mods/Cryptid/lib/overrides.lua:335,371 `local upd=Game.update; function Game:update(dt) upd(self,dt) ...`). A sibling pipeline at love.update would be the 6th wrapper and re-introduces ordering-fragility (legacy-then-pipeline). A G:draw sibling bypasses SMODS DrawStep.obj_buffer (parity B6).

THE ACTUAL SEAM: the two `for k,v in pairs(self.MOVEABLES) do ... v:move / v:update` loops at game.lua:2843 and game.lua:2849, plus the per-CardArea CardArea:align_cards (cardarea.lua:461) and CardArea:draw (cardarea.lua:320). These are the real per-frame O(N)/O(N log N) hot loops the foundation doc blames — and CRITICALLY they are NOT wrapped by SMODS or Cryptid (verified: Cryptid/SMODS wrap Game:update/Game:draw/Game:update_selecting_hand, never the MOVEABLES iteration or align_cards). The seam is therefore inside the loop body, downstream of every override, single-writer territory.

WHY IT IS A SEAM, NOT AN OVERRIDE: an override wraps and re-calls the original (the ref()-then-extend pattern at ui.lua:99 / overrides.lua:371) — order is implicit in who-calls-whom and removal leaks. The seam INSERTS a system registry between the iteration and the per-object work: game.lua:2843's `for k,v in pairs(self.MOVEABLES)` becomes `Pipeline.run('move', self.MOVEABLES, move_dt)` which iterates the ACTIVE SLICE and dispatches registered systems in explicit order. Features call `Pipeline.register{system='layout', order=210, fn=...}` — they add an ordered entry to a list, they do NOT capture-and-rewrap a function reference. Removal is one table delete; order is queryable data; no system owns the next system's invocation (kills the rewind-crash class structurally). The integration hook is one line at game.lua:2843 (move loop) and one at game.lua:320/CardArea:draw (render). It survives the override stack because it lives below it.

## render system

PROBLEM: game.lua:2843/2849 iterate ALL of self.MOVEABLES every frame; CardArea:draw + the SMODS DrawStep.obj_buffer loop (card_draw.lua) redraw every card every frame — 2 shader binds + 18 uniform sends + 2 draws per idle joker (foundation doc Root Cause A), ×325 cards = the chug.

ACTIVE-SET SYSTEM (`ecs/systems/render.lua` + `ecs/systems/activeset.lua`): ActiveSet(order 100) runs first in the move loop and sets Active.visible per card from a PURE predicate over already-existing single-writer state — visible iff (card in a drawn CardArea AND area passes CardArea:draw's own visibility gate at cardarea.lua:321-330) OR card.states.drag.is OR card.states.hover.is OR has live juice/tilt OR VT≠T (mid-ease) OR FRAME.MOVE active. Move(2843) and the render batch iterate ONLY Active.visible. Settled off-screen/idle cards are skipped.

PIXEL-IDENTICAL INVALIDATION RULES (the parity-skeptic's demand, exhaustive — a card is RE-RENDERED, never cache-served, when ANY is dirty): VT.x/VT.y/VT.r/VT.scale delta (read from moveable.lua:414-454, the eased values — this is why VT MUST be a component, B3); juice (moveable.lua:270); tilt; states.hover/click/drag; debuff/greyed; highlight (cardarea.lua:556); edition/sticker/seal (these animate -> the card is FORCE-active every frame, never cached); sprite/atlas swap (card.lua set_sprites); ability/value change; shadow_parrallax.x (cardarea.lua:559, depends on G.ROOM.T.w -> resize invalidates all); GRAPHICS.shadows setting; FBO size on resize/fold. The static-composite FBO (foundation #4) caches ONLY the shadow+center layers of a card whose entire invalidation key is unchanged; edition layers always redraw over it.

ORDERING RULE: the render system MUST dispatch through SMODS.DrawStep.obj_buffer (the sorted order array at card_draw.lua post_inject_class), NOT a custom draw — it iterates the SAME obj_buffer steps in the SAME sorted order, only restricting the card SET to the active slice and the cached cards to the edition pass. This closes parity gap B6 (missing-visual blindness of a score-only oracle). LAZY_SHADER nil-reset refinement (foundation #3, engine/sprite.lua:123 already half-done — the nil-reset is removed, callers self-reset) collapses the shadow->center same-dissolve ping-pong.

## effect system

PROBLEM: SMODS calculate_context/eval_card (utils.lua) allocates ~340 throwaway tables/hand (J=8) + ~50 context literals + ~170 {eval} wrappers; GC piles onto scoring (foundation Root Cause).

REGISTERED-EFFECT POOLED-CONTEXT DISPATCH (`ecs/systems/effect.lua`), SCORING-BYTE-IDENTICAL: (1) `SMODS._ctx_pool` — a 32-slot pre-allocated LIFO stack of context tables; `acquire_context()` pops + clears-in-place, `release_context()` pushes. LIFO is mandatory because it matches the push/pop nesting of nested calculate_context calls so identity checks (`context == X`) still hold. (2) A joker->context SUBSCRIPTION INDEX: each registered effect declares which context types it reacts to; a context dispatch visits ONLY subscribed jokers instead of scanning all G.jokers. (3) Cached per-joker derived values, invalidated on ability/value change. API: features call `Effect.register{key='j_foo', on={'individual','before','after'}, fn=function(ctx,card) ... end}` — replacing the monkeypatch where every joker injects via SMODS.calculate. The pipeline owns the dispatch loop; jokers own only their fn.

INVALIDATION/ORDERING RULES (parity-skeptic A1-A13): (a) dispatch order is the EXISTING SMODS joker order (G.jokers.cards order + SMODS sort), reproduced exactly — RNG-seeded jokers (chili_pepper) and cross-joker copy (brainstorm/blueprint) read neighbors in the same order, so seeded draws and copy targets are bit-identical. (b) The pooled context is released ONLY after the full nested dispatch unwinds (LIFO release matches acquire) — a global-state monkey-patch joker that stashes a context reference across calls would alias a recycled table, so any effect that ESCAPES its context (stores it past its fn return) is BANNED from the pool and gets a fresh table — detected by the harness comparing GC-bytes AND scores. (c) end-of-round self-destruct (chili_pepper) fires in the same event-queue slot. (d) big-num/OmegaNum/Talisman values pass through unchanged (no numeric coercion in the pool path). WHAT IT STRANGLES: the per-joker `local ref = SMODS.calculate_context; function SMODS.calculate_context...` monkeypatch surface in mods, and the {eval}-wrapper allocation in utils.lua eval_card.

## parity oracle

HARNESS (`test/oracle-check.sh`, already scaffolded on disk; `test/score-oracle.sh` runner; `test/score-oracle-baselines.txt` 10 Cryptid baselines covering all 13 archetypes per commit 76f4175). BOOT: deterministic nix-shell + xvfb-run love over build/game/main.lua (regen-dump style, headless). DRIVER: per baseline a forced seed + forced hand + forced joker loadout (ORACLE_JOKERS env var, commit bf061b5). CAPTURE: the played-hand score read from game state, emitted as golden record `seed -> [hand scores]`. Each run copies build/score-oracle/check/oracle.log out before the next (it is overwritten per invocation). EXIT: 0 iff every baseline matches byte-for-byte, 1 on any mismatch.

EXTENSION FOR THIS REWRITE (two new gates the score oracle alone cannot see — closes parity B1-B8/B6): (1) ACTIVE-SET INVARIANT gate — a debug assertion mode where ActiveSet computes Active.visible AND a full-set render runs in parallel for N frames; any card drawn in full-set but excluded from active-set is a parity failure (catches a missing invalidation rule). (2) DRAW-CALL TRANSCRIPT gate — capture the ordered sequence of (card, DrawStep.key, shader, layer) tuples emitted per frame for a fixed scripted sequence; golden-compare. This makes a MISSING VISUAL (the score-oracle's blind spot) a hard failure. SEEDS: the existing baseline seeds plus one resize event and one drag/hover event per scripted run (exercises the FBO-size and hover invalidation rules). GOLDEN FORMAT: `seed -> {scores:[...], drawcalls:sha256, activeset_ok:bool}`.

3. ## top risks + resolving rules

1. SEAM (seam-skeptic): a pipeline sibling at love.update/G:update is the 6th wrapper atop ui.lua:98 + Cryptid overrides.lua:335/371 and re-introduces legacy-then-pipeline ordering fragility. RULE: the seam is game.lua:2843 (MOVEABLES move loop) + CardArea:draw/align_cards — hot loops that are NOT wrapped by SMODS/Cryptid (verified), below the override stack. Features Pipeline.register an ordered entry; they never capture-and-rewrap a function ref.

2. OWN-vs-VIEW (seam-skeptic fatal core, code.lua:2577): composition can only kill the rewind crash by OWNING data, but coexistence requires VIEWING it; swap-over-views = use-after-free, own-data = T dual-writer (BREAK 3). RULE: split components by writer-count. Core OWNS only fields with zero legacy writers (Active). VT/Sprite are VIEWED read-only (one writer each). T is CONTESTED (two writers: align_cards + move) and stays legacy-owned through the whole migration — Layout writes the identical T, never owns it.

3. VT ABSENT FROM COMPONENTS (parity-skeptic B3): the game draws the EASED VT (moveable.lua:404-454) behind the FRAME.MOVE guard, not T. A render system reading T diverges visibly. RULE: VT.{x,y,r,scale} is a first-class VIEWED-RO component; all render-batch and active-set invalidation keys read VT, never T. T≠VT means mid-ease means force-active.

4. DRAW BYPASS (parity-skeptic B6): a custom render bypasses SMODS.DrawStep.obj_buffer (card_draw.lua) and a score-only oracle is blind to missing visuals. RULE: Render(500) dispatches through the SAME sorted obj_buffer in the SAME order, restricting only the card SET (active slice) and the cached layers (edition-only over FBO). The draw-call-transcript oracle gate sha256-compares the (card,step,shader,layer) tuple stream, making a missing visual a hard failure.

5. EFFECT IDENTITY/ORDER (parity-skeptic A1-A13): a context pool aliases recycled tables; reordered dispatch breaks RNG-seeded (chili_pepper), cross-joker copy (blueprint/brainstorm), and global-state monkeypatch jokers. RULE: 32-slot LIFO pool (release only on full unwind, matching nested acquire) preserves context==X identity; dispatch order = existing SMODS joker order exactly; any effect that escapes its context past fn-return is banned from the pool and gets a fresh table; big-num/Talisman values pass through uncoerced. Verified by byte-identical score oracle + GC-bytes A/B.

6. MIGRATION SHIPPABILITY (migration-skeptic): a step that adds a system without deleting real patch surface is not strangling. RULE: every step deletes named surface and is independently parity-green — Step1 deletes the full-MOVEABLES iteration, Step2 the per-frame settled-card shadow/center binds, Step3 the align_cards O(N log N) sort+rank + count_property 2×O(N), Step4 the per-joker SMODS.calculate_context monkeypatch + eval-wrapper allocation. First PR (ActiveSet) touches only the zero-writer Active component, making the dual-writer hazard structurally impossible to hit.
